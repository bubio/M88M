# CI

## Artifact names

Build artifacts use this naming pattern:

```text
m88m-<platform>-<arch>-<format>
```

Use lowercase names. Use `x64`, `x86`, `arm64`, `armhf`, or `universal` for
the architecture field. Use the package/archive type for the format field.

Current artifact names:

```text
m88m-linux-x64-deb
m88m-linux-x64-rpm
m88m-linux-x64-appimage
m88m-linux-arm64-deb
m88m-linux-arm64-rpm
m88m-linux-arm64-appimage
m88m-linux-armhf-deb
m88m-windows-x64-zip
m88m-windows-x86-zip
m88m-windows-arm64-zip
m88m-macos-universal-app
m88m-macos-universal-dmg
m88m-freebsd-x64-pkg
```

Release asset filenames should match the same platform, architecture, and
format fields where possible:

```text
m88m-linux-x64.AppImage
m88m-linux-arm64.AppImage
m88m-windows-x64.zip
m88m-windows-x86.zip
m88m-windows-arm64.zip
m88m-macos-universal.dmg
```

DEB, RPM, and FreeBSD pkg filenames are generated by CPack or the platform
packaging script, but their artifact names still follow the same rule.

## Workflow triggers

Platform workflows use `paths` instead of cross-platform `paths-ignore`.
Each workflow should list only files that can affect that platform build:

- shared build inputs: `CMakeLists.txt`, `LICENSE`, `assets/**`, `src/**`
- platform build script, when one exists
- the platform workflow itself
- `.github/workflows/publish-release-assets.yml`

This keeps documentation-only changes and unrelated platform workflow changes
from starting unnecessary builds.

## Release publishing

Platform workflows build and upload artifacts with read-only repository
permissions. Release asset publishing is delegated to
`.github/workflows/publish-release-assets.yml`, which is a reusable workflow
with `contents: write` only for the upload job.

The reusable workflow downloads artifacts from the current run, filters them by
filename glob, and uploads them with `gh release upload --clobber`.
